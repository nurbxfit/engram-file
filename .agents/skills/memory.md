# Memory System - How It Works

## What Is It
A two-layer file-based Memory System at the project root.
- `memory/index.jsonl` is the index вЂ” fast, scannable, low token cost.
- `memory/details/<module>/*.md` are detail documents вЂ” loaded only when needed.

## When to Use Memory

### At Session Start
1. Read `memory/index.jsonl` fully.
2. Run `python scripts/engram.py digest --task "<your task>"` for relevance-filtered summary.
3. For each relevant entry that has a `details` field, read that markdown file.
4. Use this context before writing any code or making any decisions.

### During a Session
Consult memory when you:
- Are about to make an architectural decision.
- Are unsure if something was decided before.
- Notice a pattern that contradicts past decisions.
- Encounter an error type you may have seen before.

## What Should Be Saved as Memory

Save when:
- A decision is made that will affect future sessions (e.g. "Use JWT, avoid sessions")
- A bug pattern is found that could recur (e.g. "circular dep when importing from index")
- A convention is established (e.g. "always wrap DB calls in service layer")
- A failed approach is ruled out (e.g. "tried X, caused Y problem, avoid it")

Do NOT save:
- Things already in `AGENTS.md` or docs.

Prompt the user before saving:
- One-off fixes with no future relevance.
- Temporary experiments.

## Entry Format

```jsonl
{
  "id": "auto-generated-slug-timestamp",
  "ts": "2026-05-19",
  "type": "decision|error|pattern|convention",
  "topic": "short-topic-slug",
  "tags": ["tag1", "tag2"],
  "note": "one line summary вЂ” what to remember",
  "status": "active|superseded",
  "module": "auth|db|api|...",
  "layer": "frontend|backend|infra",
  "details": "filename.md"
}
```

**Types:**
- `decision` вЂ” architectural or design choice made.
- `error` вЂ” bug pattern found and resolved.
- `pattern` вЂ” reusable approach that worked well.
- `convention` вЂ” team/project style rule.

## Using the CLI

All additions and validation go through `scripts/engram.py`:

```bash
# Session start вЂ” get digest filtered by task
python scripts/engram.py digest --task "implement JWT auth"

# Add a new entry (validates schema automatically)
python scripts/engram.py add \
  --type decision \
  --topic jwt-auth \
  --tags auth,security \
  --note "Use JWT, avoid sessions" \
  --module auth

# Query active entries
python scripts/engram.py query --topic jwt
python scripts/engram.py query --module auth
python scripts/engram.py query --tags db

# Archive a superseded entry
python scripts/engram.py archive --topic old-sessions

# Validate all entries
python scripts/engram.py validate

# Regenerate by-module/ views
python scripts/engram.py sync-views
```

## Detail Files

Detail `.md` files live in `memory/details/<module>/` using the format:
```
{topic}-{subtopic}-{type}.md
```

Examples:
```
memory/details/auth/jwt-spec-guards.md
memory/details/db/connection-pooling-pattern.md
memory/details/api/rate-limit-decision.md
```

## How to Update an Entry

- If a decision is reversed, use `python scripts/engram.py archive --topic <topic>` to move it to archive.jsonl вЂ” never delete.
- Add a new entry with `--supersedes <id>` referencing what it supersedes.
- Update the linked `.md` file if the details change, or create a new one.

## Token Awareness
- index.jsonl entries are cheap вЂ” read all of them at session start.
- Detail `.md` files cost tokens вЂ” only load when directly relevant.
- Never load all detail files speculatively.
- Use `engram digest --task <task>` to avoid scanning entries that don't match.
